# Activity Log / Audit Trail — Backend Plan

Read-only audit trail of significant board/card actions: who did what, when,
with structured `meta` (JSONB) so the frontend renders human-readable lines with
no extra lookups. Two reads: a per-card timeline and a per-board feed
(paginated). No write endpoint — rows are system-generated by instrumenting the
existing mutations. Permission to read = board `view` (resolved via the same
card/board chain the mutations already use).

Mirror `features/comment` + `features/assignee` patterns most closely:
`*.router.ts` / `*.service.ts` / `*.repo.ts` + `test/<endpoint>.spec.ts`, Kysely,
tRPC `protectedProcedure`, Zod from `shared`, OpenAPI `.meta`, superjson.

**Naming (decided, audit B2):** router export `activityRouter`, registered under
the SINGULAR key `activity` in `trpc/router.ts`, FE calls `trpc.activity.*`.
Every other feature uses a plural key (`cards`, `boards`, `comments`,
`assignees`) — `activity` is the deliberate exception (reads as a feed name).
Backend and frontend plans MUST stay in sync on this key.

## Key decisions (decided)

- **After-commit, NOT same transaction.** Mutations today do NOT wrap their work
  in a Kysely transaction — each service runs sequential `db.*` calls on the
  shared `appDb` (e.g. `card.service.deleteCard` `card.service.ts:192`,
  `assignee.service.assign` `assignee.service.ts:79`). The recorder is called
  AFTER the originating write succeeds, on the same `db`, in the same request,
  and is wrapped in try/catch so a recording failure is logged and swallowed —
  it MUST NOT fail the user action. Justification: a dropped audit row is far
  less harmful than failing a successful card move/delete; consistency of the
  audit log is best-effort by design (this is an append-only side log, not part
  of the domain invariant). Putting it in the same transaction would (a) require
  threading a transaction through every service that does not have one today
  (large change) and (b) make a logging bug roll back the user's real action —
  the opposite of the requirement.
- **Card delete keeps the audit row.** `activities.card_id` is
  `ON DELETE SET NULL`; the recorder always stores `meta.cardTitle` (and
  `meta.cardId` for the deleted card) so a `card.deleted` row remains readable in
  the board feed after the card is gone. `activities.board_id` is
  `ON DELETE CASCADE` — deleting a board removes its activity (a deleted board has
  no feed to show, and retaining orphaned audit rows for a vanished board adds no
  value). `actor_id` is `ON DELETE SET NULL` so deleting a user does not erase
  history; the feed falls back to a placeholder handle when the actor is gone.
- **Event taxonomy = 22 types** (single shared enum, see §2). Grouped:
  card (5), label (2), assignee (2), due (2), cover (1), comment (1),
  attachment (2), checklist (2), checklist-item (3), board-member (2).
- **Pagination = limit/offset** for the board feed, matching the existing list
  pattern (`backup.repo.listRuns` `backup.repo.ts:144` uses
  `.limit().offset()`). Card timeline is unpaginated (a single card's history is
  small; returns all rows newest-first) — keep it simple and consistent with
  `comment.listComments` returning a full list.
- **Actor handle resolution = one batched query**, exactly like
  `comment.service.buildComments` (`comment.service.ts:71`): collect distinct
  `actor_id`s, one `users` `where id in (...)`, map to `nameFromEmail` handle. No
  N+1. `PublicUser` has no name/avatar (`auth.schema.ts`), so the actor is shown
  as the email local-part handle (mirror `comment.service.nameFromEmail`
  `comment.service.ts:42`).

## 3. Recorder (`features/activity/activity.recorder.ts`)
- [x] `record(db, input)` — the single best-effort entry point. Signature:
  `record(db: Db, input: { boardId: string; cardId?: string | null; actorId: string; type: ActivityTypeValue; meta?: ActivityMeta }): Promise<void>`.
  - Body (NOTE audit B1 — `meta` MUST be `JSON.stringify`'d; passing a raw object
    to a jsonb column via node-pg sends `"[object Object]"` and corrupts the row.

    ```ts
    try {
      await db.insertInto("activities").values({
        board_id: input.boardId,
        card_id: input.cardId ?? null,
        actor_id: input.actorId,
        type: input.type,

      logger.error({ err, event: LogEvent.ActivityRecordFailed, type: input.type, boardId: input.boardId, cardId: input.cardId ?? null }, LogEvent.ActivityRecordFailed);
    }
    ```
  - NEVER throws. Uses `logger` (`../../logger.js`) like the best-effort cleanup
    blocks in `card.service.deleteCard` (`card.service.ts:201-203`) /
    `attachment.service` (`attachment.service.ts:111,128,170`).
  - Per `backend.md` "no string literal for logging event" rule: add
    `ActivityRecordFailed: "activity.record.failed"` to the `LogEvent` object in
    `config/const.config.ts` (mirror the existing entries, e.g.
    `CardReminderSent`) and reference `LogEvent.ActivityRecordFailed`.
- [x] `activity.repo.ts`:

## 6. Instrument existing mutations (each is one `record(...)` call AFTER the write)

service already has it or the column row in hand) and call the recorder. Actor is
`user.id`. `record` is best-effort, so no try/catch is needed at the call site
(the recorder swallows its own errors). Where a NAME/EMAIL is needed in `meta`
(column name, label name, assignee email), it is already loaded or one cheap
lookup — note it per task.
